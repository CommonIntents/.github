# CommonIntents-144 贡献指南
感谢您有兴趣为 CI-144 项目贡献力量！
本文档明确了项目的治理模型与完整贡献流程。

---

## 1. 治理模型
CI-144 采用**精英治理模型**，角色与职责划分清晰：

| 角色 | 核心职责 | 任职要求 |
| :--- | :--- | :--- |
| **协议维护者** | RFC 最终审批、版本发布、破坏性变更裁定 | 深入理解全部 4 项协议；2 年以上项目贡献经历 |
| **工作组成员** | RFC 投票、技术评审、标准提案评估 | 在对应协议层级具备可验证的专业能力 |
| **社区贡献者** | 缺陷反馈、文档优化、实现落地建议 | 积极参与社区讨论 |

---

## 2. RFC（意见征集）流程
**核心规范的所有变更必须通过 RFC 流程推进。**

该机制保障：
- 向后兼容性
- 理论严谨性
- 跨层一致性
- 无共识不落地破坏性变更

### 2.1 RFC 提交流程
```
┌─────────────────────────────────────────────────────────┐
│  步骤 1：起草 RFC                                        │
│  • 在 spec/drafts/ 目录下创建 Markdown 文件               │
│  • 遵循第 3 章的 RFC 模板                                 │
│  • 必须包含：设计动机、影响范围、安全分析                     │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  步骤 2：提交拉取请求                                      │
│  • 标题前缀：[RFC] 协议名: 简短标题                         │
│  • 关联相关 Issue                                        │
│  • 初始状态标记为「草稿」                                  │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  步骤 3：社区公开讨论（不少于 14 天）                        │
│  • GitHub 上公开评论周期                                  │
│  • 工作组评审以下维度的影响：                               │
│    - 向后兼容性                                          │
│    - 安全边界                                            │
│    - Milk Zen 设计哲学契合度                              │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  步骤 4：共识投票                                         │
│  • 轻微变更：2 位维护者审批通过                             │
│  • 重要变更：3 位维护者审批通过                             │
│  • 破坏性变更：种子节点 3 选 3 多重签名通过                  │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  步骤 5：合并与发布                                       │
│  • 更新协议语义化版本号                                    │
│  • 更新 README 中的状态矩阵                               │
│  • 如适用，同步发布至 IANA 注册表                           │
└─────────────────────────────────────────────────────────┘
```

### 2.2 RFC 变更分级

| 变更等级 | 定义 | 审批要求 |
| :--- | :--- | :--- |
| **轻微** | 表述澄清、文字修正、非规范性内容调整 | 1 位维护者 |
| **标准** | 新增可选字段、错误码、扩展能力 | 2 位维护者 |
| **重要** | 新增帧类型、新增动词、协议行为变更 | 3 位维护者 |
| **破坏性** | 与现有实现不兼容的变更 | 种子节点 3 选 3 多重签名 |

## 3. RFC 模板
提交 RFC 时，请遵循以下结构：

```markdown
# RFC-[编号]: [标题]

## 状态
[ ] 草稿
[ ] 评审中
[ ] 已接受
[ ] 已拒绝

## 设计动机
说明本次变更的必要性。

## 变更方案
变更的详细技术规范说明。

## 影响分析

### 向后兼容性
[ ] 完全兼容
[ ] 部分兼容（需依赖版本协商）
[ ] 破坏性变更

### 安全影响
说明相关的安全考量。

### 性能影响
说明相关的性能影响。

## 实现说明
给实现方的落地指导。

## 参考资料
- 相关 RFC
- 外部标准

## 更新日志

| 日期 | 作者 | 变更内容 |
| :--- | :--- | :--- |
| YYYY-MM-DD | 姓名 | 初始草稿 |
```

### 4.2 编码规范

| 类别 | 标准 |
| :--- | :--- |
| **规范文档语言** | 美式英语，Markdown 格式 |
| **代码示例** | 优先 Rust，兼容 Python、TypeScript |
| **命名约定** | 文件使用蛇形命名，类型使用大驼峰命名 |
| **文档要求** | 所有公开 API 必须附带文档注释 |

### 4.3 测试要求
- **单元测试**：所有校验逻辑必须覆盖
- **集成测试**：跨协议交互场景必须覆盖
- **兼容性测试**：必须通过对上一个次版本的兼容验证

## 5. 扩展开发指南
CI-144 预留了自定义实现的扩展区间：

| 协议 | 预留范围 | 用途 |
| :--- | :--- | :--- |
| BIND-19 帧类型 | `0x0F - 0xEF` | 实验性帧 |
| CAPABILITY-13 作用域 | `custom_scopes` 配置段 | 业务自定义权限 |
| INTENT-7 动词 | `x-*` 前缀 | 实验性动作 |
| INTENT-7-SECURE 错误码 | `0x07F0 - 0x07FF` | 实现自定义错误 |

**重要提示**：使用预留区间的扩展不得与未来官方标准分配产生冲突。

## 6. 版本管理策略
CI-144 遵循**语义化版本 2.0.0** 规范：

| 版本变更 | 含义 | 示例 |
| :--- | :--- | :--- |
| **主版本 (X.0.0)** | 破坏性变更 | 帧格式变更、强制字段移除 |
| **次版本 (0.X.0)** | 新增功能，向后兼容 | 新增帧类型、新增动词 |
| **修订号 (0.0.X)** | 缺陷修复、表述澄清 | 文字修正、文档更新 |

**版本协商**：BIND-19 通过零往返握手处理版本兼容。实现方必须支持同一次版本下的所有修订号。

## 7. 社区行为准则

### 7.1 行为标准
- **保持尊重**：以专业、尊重的态度对待所有贡献者
- **就事论事**：批判观点，而非攻击个人
- **严守流程**：遵循 RFC 流程，不搞捷径特例
- **面向全球**：使用清晰易懂的英文，避免习语与俚语

### 7.2 禁止行为
- 骚扰、歧视与冒犯性言论
- 为图便利绕过 RFC 流程
- 故意引入安全漏洞与后门
- 对外不实宣传协议特性

### 7.3 处罚机制
违规行为可能导致：
- 警告并要求纠正行为
- 临时禁止贡献权限
- 严重或屡犯者永久封禁
