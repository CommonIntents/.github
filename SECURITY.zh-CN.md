## 1. 安全哲学

CommonIntents-144 生态遵循**信任最小化**原则。

- **不信任提交者的自述**。每一份进入共享知识树的适配器，必须经过自动化审查与密码学签名。
- **不依赖人工审查**。安全是程序化的。AI 审查 AI，程序审查程序，Tuck 兜底。人类仅在申诉时介入——作为最高法院，而非日常警察。
- **不绑定任何平台**。共享知识树基于 IPLD DAG，CID 内容寻址。GitHub 可以消亡，Cloudflare 可以消亡，任何单一服务商可以消亡。CID 永存。知识树的入口是内容哈希，不是域名。
- **不依赖外部基础设施**。依赖总是被动的。生态必须强壮自身：任何节点都可以成为知识树的分发节点，任何人都可以独立验证适配器的签名和完整性，不需要信任任何第三方。
- **按需而动**。用户根据自身安全要求，选择从哪个知识树加载。安全策略是运行时选择，不是协议强制。

**生态自治，丝滑流淌。人工的任务是不断完善审查程序，完善审查程序是为了减少人工。**

## 2. 安全等级定义

适配器按技术能力边界分为四个等级：

| 等级 | 定义 | 能力边界 | 运行环境约束 |
|:---|:---|:---|:---|
| **Grade 0** | 纯函数 | 仅数据格式转换、Schema 映射。严禁 I/O、网络、系统调用。 | WASM 沙箱，禁用所有系统接口 |
| **Grade 1** | 受限局部 I/O | 允许读写指定的受限目录或配置文件。禁止网络访问。 | 虚拟文件系统映射沙箱 |
| **Grade 2** | 受限网络访问 | 允许 HTTPS 访问白名单中的 API 接口。禁止任意网络连接。 | 网络白名单网关过滤 |
| **Grade 3** | 系统级调用 | 允许执行 CLI 命令、操作数据库、复杂网络交互。 | 物理隔离容器，用户自行承担最终风险 |

## 3. 自治审查流水线

所有提交进入共享知识树的适配器，依次经过五步自动审查：

```
提交(源码) → AI 语义审查 → 静态 AST 审查(源码) → 受信环境编译(WASM) → 沙箱动态审查(WASM) → Tuck 自动分级签名 → 进入知识树
                                                                                              ↘ 无法判定 → 申诉队列
```

**第一层：AI 语义审查**

- 分析适配器的代码结构和 INTENT-7/CAPABILITY-13 声明
- 检测语义不一致、过度声明、隐藏行为
- **仅分析代码结构和协议声明，不读取注释、文档字符串或任何自然语言描述**
- AI 审查结果仅作为分类建议和参考权重，不得作为准入的唯一依据。第二层（AST）和第三层（沙箱）的审查结果具有更高的裁决优先级

**第二层：静态 AST 审查**

- 解析 AST，检测系统调用、网络访问、文件操作
- 网络域名白名单校验
- 文件路径越界检测
- **凭证与敏感信息扫描**：检测硬编码的高熵值字符串（疑似密钥、密码、Token）以及非安全的局域网内网 IP。一旦发现信息泄露风险，直接拒绝提交
- 通过或标记可疑

**第三层：受信环境编译**

- 在 TEE 或官方构建流水线中将审核通过的源码编译为 WASM 字节码
- 编译过程可重现，任何第三方可独立验证编译结果与源码的一致性
- 不接受提交者本地编译的二进制文件
- **资源防护**：构建流水线对提交源码设有严格的尺寸与复杂度限制，并基于提交者 INTENT-7-SECURE 证书实施动态速率限制，防止恶意高频提交耗尽公共编译资源

**第四层：沙箱动态审查**

- 在隔离环境中运行编译后的 WASM 字节码
- 主要监控系统调用序列和网络访问模式，而非完整业务逻辑：
  - 是否试图读取未声明的本地文件
  - 是否试图建立白名单之外的网络连接
  - 是否执行了非预期的子进程
- 对外部 API 的调用使用受限行为观测，不依赖完整功能集成测试
- 与 Manifest 声明行为对比，检测偏差
- 通过或标记可疑

**第五层：Tuck 最终裁决**

- 汇总前四层审查结果
- 根据生态安全策略自动分级（Grade 0-3）
- 密码学签名，纳入共享知识树
- 仅当无法自动判定时，送入申诉队列

## 4. 申诉机制

提交者对自动分级有异议时，可发起申诉。申诉是人类介入的唯一入口。

- 提交者提交申诉请求，附证据
- 人类审查者进行 Peer Review
- 审查结果更新适配器分级
- 审查结论形成先例，反哺审查规则优化

## 5. 分层信任与知识树

生态提供多层知识树，用户按需选择信任级别：

| 知识树 | 维护者 | 信任度 | 条目特征 |
|:---|:---|:---|:---|
| **官方树** | CommonIntents-144 维护者 | 最高 | 经过完整五层审查 + 签名 |
| **社区树** | 全体使用者 | 热度驱动 | 自动提交，热度排序，信任权重由使用频率和交叉验证累积 |
| **项目树** | 项目维护者 | 高于社区 | 项目自行验证和签名，服务于特定场景 |

用户根据自身安全要求选择加载策略：

- **严格模式**：仅加载官方签名条目
- **平衡模式**：官方优先 + 热度排序社区条目
- **开放模式**：全部条目，热度排序

安全策略是运行时选择。协议不强制，用户自决。

## 6. 漏洞反馈与生态自愈

任何使用者（人类或 AI）可提交漏洞报告，标记某个适配器 CID 为“有风险”。

- 标记行为的权重由标记发起者的身份可信度决定：
  - 官方签名节点（CommonIntents-144 维护者）：权重最高，即时生效
  - 已验证的长期贡献者（INTENT-7-SECURE mTLS 证书持有者）：权重中等
  - 匿名或新注册节点：权重最低，仅作为参考信号
- 单个 INTENT-7-SECURE 证书持有者在单位时间内发起的负面标记数量设有上限。若高权重节点出现高频、大范围的异常标记行为，其反馈权重自动冻结，相关适配器标记保护，强制送入申诉队列
- 当不同权重的负面标记累积超过动态阈值时，适配器自动降级或标记为不可用
- 阈值由安全策略动态调整，不由单一数值固定
- AI 审查程序根据反馈持续学习，提升审查准确率
- 人类维护者定期审核反馈数据，优化审查规则

**生态自愈。漏洞发现越快，保护越强。**

## 7. 与协议族的关系

| 协议 | 本体系如何引用 |
|:---|:---|
| **INTENT-7** | 适配器声明的意图类型，是 AI 语义审查的比对基准 |
| **CAPABILITY-13** | 适配器的 Manifest 声明业务风险等级（`securityClass`），与技术安全等级（Grade）互补 |
| **INTENT-7-SECURE** | mTLS 身份验证确保提交者身份可追溯 |
| **BIND-19** | 格式协商确保审查工具与适配器之间通信协议一致 |

**Grade 是技术安全等级**（由审查流程自动判定）：这个适配器在技术上能做什么。  
**`securityClass` 是业务风险等级**（由适配器作者在 Manifest 中声明）：这个操作在业务上有多危险。  
两者共同构成适配器的完整风险画像。

## 8. 知识树独立性

映射表知识树是 CommonIntents-144 生态的独立公共资产。它与任何特定实现的内部知识树在物理上隔离、在逻辑上独立。

- 映射表知识树仅存储 INTENT-7 意图到原生操作的翻译规则
- 不存储任何实现的私有记忆、经验原则、社会关系或用户画像
- 知识树的审查、签名、分发由本安全策略定义，不依赖任何外部系统
- 任何兼容 INTENT-7/CAPABILITY-13 的实现均可加载映射表知识树中的条目，其内部知识树的管理完全独立

## 9. 平台独立性与内容寻址

共享知识树的权威来源是内容哈希，不是任何特定服务商的域名或存储系统。

- 每个适配器在通过审查并签名后，生成唯一的 CID。CID 是适配器在生态中的永恒标识
- 知识树的权威入口是基于 IPNS 或 DNSLink 的动态指针，其最终解析指向最新、且经密码学验证的 Root CID。当 DNS 可用时，DNSLink 提供人类可读的入口。当 DNS 不可用时，IPNS 提供纯去中心化的替代路径。两者互为备份，确保入口永不断裂
- 任何兼容 IPFS 的网关、任何拥有 CID 副本的节点，都可以提供适配器的下载和验证服务
- 用户通过 CID 加载适配器，不关心文件托管在哪里。只要有一个节点保有副本，知识树就继续存在
- 生态不依赖任何单一平台。服务商可以消亡，CID 永存
- 用于验证官方签名知识树的创世公钥，硬编码锚定在客户端的二进制发行版中。公钥的轮转与撤销通过链上多签声明或客户端版本升级强制更新，确保信任根不依赖任何外部网络请求

## 10. 已知限制

本安全策略定义了单个适配器的技术安全等级和审查标准。

- 在多适配器编排环境中，数据在节点间流转可能产生复合风险（例如 Grade 1 读取敏感文件，Grade 2 联网发送数据）。本策略不覆盖跨适配器的信息流控制。
- 复合风险的检测与防护由编排引擎（如 FlowModus）负责，不在单个适配器安全审查的范围内。

## 11. 当前阶段与远期规划

**当前阶段**：

- 人类维护者审查所有 PR（人工审核）
- 人类对审查结果进行签名
- 映射表文件托管在多个独立节点上，CID 作为权威标识
- 加载映射表前验证密码学签名

**远期愿景**：

- 五层自动审查流水线全面运行
- Tuck 自动分级签名
- IPLD 共享知识树通过 IPFS 网络分发，CID 寻址
- 共享知识树不接受用户本地编译的二进制文件。用户提交源代码，由官方构建流水线在受信环境中编译并签名，实现可重现构建
- 签名密钥托管于硬件安全模块（HSM）或可信执行环境（TEE），签名过程需要多签或门限签名机制保护
- 动态审查环境隐藏虚拟化特征，防止沙箱嗅探。沙箱支持加速时间轴仿真，检测时间锁定的延迟后门
- 宿主解释器进程通过操作系统级安全机制加固，确保即便 WASM 引擎本身存在漏洞，也无法穿透到宿主系统
- 人类仅处理申诉和规则优化
