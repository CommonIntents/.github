# 致谢与技术溯源
本文档详细说明 CommonIntents-144（CI-144）协议族的设计参考来源、技术对比与归属说明。

---

## 1. 设计灵感与前置技术参考
CI-144 站在巨人的肩膀上构建。我们由衷感谢以下开放标准与项目为设计提供的启发。

### 1.1 传输层设计灵感

| 标准 | 设计借鉴 | 我们的创新点 |
| :--- | :--- | :--- |
| **HTTP/2 (RFC 7540)** | 基于帧的多路复用、流优先级机制 | • 精简的 8 字节固定帧头（对比 9 字节）<br>• 256 路通道 + 4 级优先级队列<br>• 零往返 UDS 隐式协商 |
| **QUIC (RFC 9000)** | 连接迁移、独立流设计 | • 本地优先设计，针对 UDS 深度优化<br>• 无 HTTP 语义的纯二进制帧复用 |

### 1.2 安全层设计灵感

| 标准 | 设计借鉴 | 我们的创新点 |
| :--- | :--- | :--- |
| **TLS 1.3 (RFC 8446)** | 双向认证、现代密码套件 | • 强制双向 TLS（无可选客户端认证模式）<br>• 仅允许 2 套高强度密码套件<br>• L0 基因锁证书钉扎机制 |
| **SO_PEERCRED (POSIX)** | 本地进程身份校验 | • UDS 场景下零毫秒开销校验<br>• 强制 UID/GID 严格匹配 |
| **PKCS#11 / TPM** | 硬件安全模块集成 | • 标准化密钥管理服务接口支持<br>• 凭证隔离边界设计 |

### 1.3 能力层设计灵感

| 标准 | 设计借鉴 | 我们的创新点 |
| :--- | :--- | :--- |
| **OAuth 2.0 (RFC 6749)** | 基于能力的访问令牌 | • 采用 Ed25519 签名（对比 HMAC-SHA256）<br>• TOML 配置格式（对比作用域字符串）<br>• 内置人机共识机制 |
| **DNSSEC (RFC 4033)** | 签名链校验机制 | • 基因锁签名层级体系<br>• 类 GPG 密钥轮换协议 |
| **OPA / Casbin** | 代码化策略、属性级访问控制 | • 声明式权限映射<br>• 带签名的动态 TOML 配置 |

### 1.4 语义层设计灵感

| 标准 | 设计借鉴 | 我们的创新点 |
| :--- | :--- | :--- |
| **JSON-RPC 2.0** | 远程过程调用模式 | • AI 原生动词体系（FETCH/WRITE_NODE/TENTACLE）<br>• 强制集成 W3C 追踪上下文<br>• HXR-L3 零拷贝内存对齐 |
| **W3C Trace Context** | 分布式追踪标准 | • 与 TLS 会话原生绑定<br>• 跨协议边界的因果链路追踪 |
| **OpenAPI** | API 规范格式 | • 面向智能体的意图优先设计<br>• 执行记录与内存的双向映射 |

## 2. 技术对比矩阵

### 2.1 BIND-19 vs HTTP/2 vs WebSocket

| 特性 | BIND-19 | HTTP/2 | WebSocket | 优势 |
| :--- | :--- | :--- | :--- | :--- |
| **帧头大小** | 8 字节固定 | 9 字节固定 | 2-14 字节变长 | 解析更简单，长度确定 |
| **多路复用** | 256 路通道、4 级队列 | 流机制、无优先级队列 | 不支持 | 更细粒度的 AI 任务调度 |
| **版本协商** | 零往返 UDS 隐式协商 | ALPN 显式协商 | 不支持 | 本地握手速度更快 |
| **错误处理** | 统一错误码体系 | 独立错误帧 | 仅支持关闭帧 | 错误链路全程一致 |
| **治理机制** | IANA 风格注册表 | HTTP/2 工作组 | W3C | 开放、标准化的扩展机制 |

### 2.2 CAPABILITY-13 vs OAuth 2.0 vs OPA

| 特性 | CAPABILITY-13 | OAuth 2.0 | OPA (Rego) | 优势 |
| :--- | :--- | :--- | :--- | :--- |
| **令牌格式** | Ed25519 签名 | JWT (HMAC/RS256) | 无（策略引擎） | 抗量子攻击，安全性更强 |
| **配置方式** | TOML + 数字签名 | JSON 作用域字符串 | Rego 领域语言 | 人类可读，带密码学签名保障 |
| **共识机制** | 内置人机共识 | 依赖外部系统 | 依赖外部系统 | 原生支持人工在环审批 |
| **密钥轮换** | 内置类 GPG 机制 | 依赖外部系统 | 无 | 安全、原子化的密钥切换 |
| **吊销机制** | 原子事务生效 | CRL/OCSP 轮询 | 无 | 立即生效，无需轮询 |

### 2.3 INTENT-7 vs JSON-RPC vs GraphQL

| 特性 | INTENT-7 | JSON-RPC 2.0 | GraphQL | 优势 |
| :--- | :--- | :--- | :--- | :--- |
| **意图表达** | AI 原生动词体系 | 方法名调用 | 查询/变更模式 | 更贴合 AI 认知模型 |
| **追踪能力** | 原生 W3C 追踪上下文 | 无 | 无 | 完整因果链路可追溯 |
| **内存对齐** | HXR ↔ L3 零拷贝 | 无 | 无 | 执行与存储直接映射 |
| **校验机制** | 强制 JSON Schema 校验 | 可选校验 | Schema 定义 | 默认开启严格校验 |
| **错误处理** | BIND-19 统一错误码 | JSON-RPC 自有错误 | GraphQL 自有错误 | 错误链路全程统一 |

### 2.4 INTENT-7-SECURE vs TLS 1.3 vs SPIFFE

| 特性 | INTENT-7-SECURE | TLS 1.3 (双向) | SPIFFE/SPIRE | 优势 |
| :--- | :--- | :--- | :--- | :--- |
| **密码套件** | 仅保留 2 套最强套件 | 多套可选 | 无相关定义 | 无弱密码，审计更简单 |
| **证书钉扎** | 强制开启（基因锁） | 可选 | SVID 轮换机制 | 防范证书颁发机构被攻破的攻击 |
| **本地安全** | SO_PEERCRED（零毫秒开销） | UDS 上仍跑双向 TLS | Unix 套接字 + 双向 TLS | 本地信任场景零额外开销 |
| **凭证隔离** | 边缘网关 + 密钥管理服务 | 应用层自行处理 | 密钥存储 | 应用全程无法接触明文凭证 |
| **会话绑定** | 追踪上下文与 TLS 绑定 | 仅会话 ID 绑定 | 无 | 密码学层承载业务上下文 |

## 4. 原创贡献
CI-144 提出了以下创新设计概念：

1. **AI 原生意图语言**：专为 AI 认知模式设计的动词体系（FETCH/WRITE_NODE/TENTACLE）
2. **零拷贝内存对齐**：HXR 执行记录直接映射至 L3 情景记忆
3. **本地零信任校验**：通过 SO_PEERCRED 实现零毫秒开销的 UID/GID 身份校验
4. **内置人机共识**：能力层原生集成人工在环审批机制
5. **统一错误链路**：所有协议错误均通过 BIND-19 错误帧统一传递
6. **基因锁签名层级**：类 DNSSEC 的签名链体系，用于权限映射校验
