## 1. 安全设计理念
CI-144 基于**零信任架构**构建，遵循以下核心原则：

| 设计原则 | 落地实现 |
| :--- | :--- |
| **1. 默认不信任任何对象** | 所有连接必须通过双向 TLS（公网）或 SO_PEERCRED（本地）完成身份认证 |
| **2. 最小权限原则** | 默认不授予任何权限；所有权限必须显式声明并通过审批 |
| **3. 纵深防御体系** | 安全防护覆盖四层：语义层、能力层、传输层、安全层 |
| **4. 拒绝隐匿式安全** | 所有密码学算法均为公开标准（TLS 1.3、Ed25519） |
| **5. 全链路可审计** | 所有事务均可通过 W3C 追踪上下文标准完成溯源 |

## 3. 安全架构
CI-144 实现了四层安全防护模型：
```
┌─────────────────────────────────────────────────────────┐
│ 第四层：语义安全（INTENT-7）                               │
│   • JSON Schema 合法性校验                               │
│   • 意图动词白名单机制                                     │
│   • 元数据完整性校验                                      │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ 第三层：能力安全（CAPABILITY-13）                          │
│   • 权限映射校验                                          │
│   • Ed25519 签名验签                                     │
│   • 高风险操作人机共识审批                                 │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ 第二层：传输安全（BIND-19）                                │
│   • 帧完整性校验和                                        │
│   • 基于序列号的防重放机制                                 │
│   • 版本协商降级防护                                      │
└─────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────┐
│ 第一层：密码学安全（INTENT-7-SECURE）                      │
│   • 双向 TLS 1.3（公网环境）                              │
│   • SO_PEERCRED（本地 Unix 域套接字）                     │
│   • 证书钉扎（L0 基因锁）                                  │
│   • 凭证隔离（边缘网关 + 密钥管理服务）                      │
└─────────────────────────────────────────────────────────┘
```

### 3.1 第一层：密码学安全

| 安全机制 | 遵循标准 | 设计用途 |
| :--- | :--- | :--- |
| **双向 TLS 1.3** | RFC 8446 | 双向身份认证、数据机密性与完整性 |
| **密码套件** | TLS_AES_256_GCM_SHA384、TLS_CHACHA20_POLY1305_SHA256 | 高强度加密、带关联数据的认证加密 |
| **证书钉扎** | L0 基因锁内存储 SHA-256 指纹 | 防范证书颁发机构被攻破导致的中间人攻击 |
| **SO_PEERCRED** | POSIX.1-2001 | 本地身份校验（零毫秒开销） |
| **密钥轮换** | Ed25519、原子事务 | 前向保密、密钥泄露后的恢复能力 |

### 3.2 第二层：传输安全

| 安全机制 | 说明 |
| :--- | :--- |
| **帧完整性校验** | 每个 BIND-19 帧均携带校验和 |
| **防重放攻击** | 每个通道内序列号单调递增 |
| **版本协商** | 仅接受双方共同支持的最高版本 |

### 3.3 第三层：能力安全

| 安全机制 | 说明 |
| :--- | :--- |
| **权限映射** | 通过 `capability_mapping.toml` 定义允许的能力范围 |
| **签名验签** | 使用 Ed25519 签名验证权限映射的真实性 |
| **人机共识** | 高风险操作需人工在环审批 |
| **密钥轮换** | 类 GPG 密钥轮换机制，具备回滚防护 |

### 3.4 第四层：语义安全

| 安全机制 | 说明 |
| :--- | :--- |
| **结构校验** | 所有 INTENT-7 载荷均需通过 JSON Schema 验证 |
| **动词白名单** | 仅接受已定义的动作类型 |
| **追踪上下文** | W3C `traceparent` 与 TLS 会话绑定 |

## 4. 威胁模型

### 4.1 预设攻击者能力

| 攻击能力 | 对应缓解方案 |
| :--- | :--- |
| **网络报文嗅探** | 双向 TLS 加密全部流量 |
| **报文注入** | 双向 TLS 提供完整性保护 |
| **重放攻击** | BIND-19 序列号机制防范重放 |
| **中间人攻击（MITM）** | 带证书钉扎的双向 TLS |
| **证书颁发机构被攻破** | L0 基因锁内的证书钉扎机制 |
| **本地权限提升** | SO_PEERCRED 校验进程 UID/GID |
| **凭证窃取** | 凭证隔离存储于边缘网关密钥管理服务 |

### 4.2 不在防护范围内

| 攻击类型 | 原因说明 |
| :--- | :--- |
| **物理设备被攻破** | 超出范围；默认假设硬件环境安全 |
| **侧信道攻击** | 通过常量时间实现缓解（如适用） |
| **拒绝服务攻击** | 由应用层实现限流防护 |

## 5. 漏洞上报

### 5.1 上报流程
**请勿**在公开的 GitHub Issue 中提交安全漏洞。

请按以下方式上报：
1. **邮件**：将详细信息发送至 `security@commonintents.org`
2. **加密**：使用我们的 PGP 公钥加密内容（指纹见 L0 基因锁）
3. **请包含以下信息**：
   - 受影响的协议与版本
   - 漏洞描述
   - 复现步骤
   - 潜在影响
   - 修复建议（如有）

### 5.2 响应时间线

| 里程碑 | 目标时长 | 最长期限 |
| :--- | :---: | :---: |
| **首次确认收到** | 24 小时 | 48 小时 |
| **严重等级评估** | 72 小时 | 7 天 |
| **严重漏洞补丁** | 14 天 | 30 天 |
| **一般漏洞补丁** | 30 天 | 60 天 |
| **公开披露** | 补丁发布后 | 90 天 |

### 5.3 严重等级评定
我们采用 [CVSS v3.1 通用漏洞评分系统](https://www.first.org/cvss/) 进行严重等级评估：

| 等级 | CVSS 分值范围 | 示例 |
| :---: | :--- | :--- |
| **严重** | 9.0 - 10.0 | 无需认证的远程代码执行 |
| **高危** | 7.0 - 8.9 | 认证绕过、私钥泄露 |
| **中危** | 4.0 - 6.9 | 权限边界突破 |
| **低危** | 0.1 - 3.9 | 信息泄露、拒绝服务 |

## 6. 密码学密钥管理

### 6.1 密钥类型

| 密钥类型 | 算法 | 用途 | 轮换周期 |
| :--- | :--- | :--- | :--- |
| **L0 创建者密钥** | Ed25519 | 权限映射签名 | 建议 90 天 |
| **TLS 证书** | X.509 v3，搭配 Ed25519/ECDSA | 双向 TLS 认证 | 建议 1 年 |
| **追踪 ID** | 128 位随机值 | 事务追踪 | 单次事务 |

### 6.2 密钥轮换流程
密钥轮换遵循 **CAPABILITY-13 密钥轮换协议**：
1. **生成新密钥**：创建新的 Ed25519 密钥对
2. **签名事务**：生成由旧密钥签发的密钥轮换事务
3. **多重签名**：需要种子节点 3 选 3 多重签名确认
4. **原子替换**：所有映射使用新密钥重新签名
5. **事件通告**：通过 BIND-19 控制帧发布轮换事件

### 6.3 密钥泄露应急响应
若私钥发生泄露：
1. **立即处置**：通过紧急事务吊销已泄露密钥
2. **短期方案**：启用备份密钥承担签名职责
3. **长期方案**：完成完整的密钥对轮换
4. **事件通知**：通过全渠道发布通告（GitHub、邮件等）

## 7. 实现方安全最佳实践

### 7.1 传输层
```rust
// 示例：BIND-19 帧校验
fn validate_frame(frame: &Frame) -> Result<(), SecurityError> {
    // 1. 校验版本兼容性
    if frame.version != BIND_19_VERSION {
        return Err(SecurityError::VersionMismatch);
    }
    
    // 2. 校验校验和
    if !frame.verify_checksum() {
        return Err(SecurityError::IntegrityCheckFailed);
    }
    
    // 3. 校验序列号（防重放）
    if frame.sequence_id <= self.last_sequence_id[frame.channel_id] {
        return Err(SecurityError::ReplayDetected);
    }
    
    Ok(())
}
```

### 7.2 安全层
```rust
// 示例：双向 TLS 配置
fn configure_mtls(config: &mut ServerConfig) {
    // 1. 设置最低 TLS 版本为 1.3
    config.set_min_proto_version(ProtocolVersion::TLSv1_3);
    
    // 2. 限制可用密码套件
    config.set_cipher_suites(&[
        CipherSuite::TLS_AES_256_GCM_SHA384,
        CipherSuite::TLS_CHACHA20_POLY1305_SHA256,
    ]);
    
    // 3. 配置客户端证书校验
    config.set_client_certificate_verifier(
        ClientCertificateVerifier::new()
            .require_client_cert(true)
            .add_trusted_fingerprint(gene_lock_fingerprint),
    );
}
```

### 7.3 凭证隔离
```rust
// 示例：边缘网关凭证注入
async fn inject_credentials(
    request: &mut Request,
    identity_label: &str,
) -> Result<(), SecurityError> {
    // 1. 解析身份标签（命名空间/标识@实例ID）
    let label = IdentityLabel::parse(identity_label)?;
    
    // 2. 从密钥管理服务获取解密后的凭证
    let credential = kms.retrieve_credential(&label).await?;
    
    // 3. 注入 HTTP 请求头
    request.headers_mut().insert(
        HeaderName::from_static("authorization"),
        HeaderValue::from_str(&format!("Bearer {}", credential.token))?,
    );
    
    Ok(())
}
```

## 9. 安全更新策略

### 9.1 补丁发布条件
满足以下条件时，将发布安全补丁：
- 漏洞已上报并确认存在
- CVSS 评分 ≥ 4.0（中危及以上）
- 野外已出现可利用的攻击代码
- 漏洞影响当前受支持的版本

### 9.2 向后兼容性
安全补丁**不会**破坏向后兼容性，除非万不得已。

若安全修复必须引入破坏性变更：
- 将以新的主版本号发布
- 配套提供迁移指南
- 对上一个主版本提供 6 个月的过渡支持期
